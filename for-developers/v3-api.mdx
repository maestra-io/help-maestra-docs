## Operation Invocation Methods

<CodeGroup>

```http Synchronous
POST /v3/operations/sync?endpointId={endpointId}&operation={operation}&deviceUUID={deviceUUID} HTTP/1.1
Host: api.maestra.io
Accept: application/xml
Content-Type: application/xml
Authorization: SecretKey {Secret Key}
X-Customer-IP: {IP of customer device}
```

```http Asynchronous
POST /v3/operations/async?endpointId={endpointId}&operation={operation}&deviceUUID={deviceUUID} HTTP/1.1
Host: api.maestra.io
Accept: application/xml
Content-Type: application/xml
Authorization: SecretKey {Secret Key}
X-Customer-IP: {IP of customer device}
```

</CodeGroup>

## EndpointId

- Unique ID of site/store/mobile app/etc.
- The value can be obtained from the manager.

## Authorization header

- `Authorization` can be required, depending on operation type.
- Check with the manager if it's mandatory.
- Each EndpointId corresponds to its own secretKey; check the value with the manager.

## Operation

- Operation name at Maestra. Each type of site/store/mobile app action has a corresponding operation.
- Operation list is set up internally.
- Request/response body format depends on operation type.

## DeviceUUID

- Check with the manager if it's mandatory.
- At the website: pass DeviceUUID from the `maestraDeviceUUID` cookie.
- At Android KitKat and higher: pass [Advertising Id](https://support.google.com/googleplay/android-developer/answer/6048248)
- At iOS 6 and higher: pass [IDFA](https://developer.apple.com/reference/adsupport/asidentifiermanager)

## Error Handling

```xml Success
<result>
  <status>Success</status>
  {data returned by the operation}
</result>
```

```xml Validation error
<result>
    <status>ValidationError</status>
    <validationMessages>
        <validationMessage>
            <message>Incorrect last name</message>
            <location>/customer/lastName</location>
        </validationMessage>
        <validationMessage> ...</validationMessage>
    </validationMessages>
</result>
```

```xml Incorrect operation invocation
<result>
    <status>ProtocolError</status>
    <errorMessage>{error text}</errorMessage>
    <errorId>{error UUID}</errorId>
    <httpStatusCode>{error HTTP code}</httpStatusCode>
</result>
```

```xml Maestra unavailable
<result>
    <status>InternalServerError</status>
    <errorMessage>{error text}</errorMessage>
    <errorId>{error UUID}</errorId>
    <httpStatusCode>{error HTTP code}</httpStatusCode>
</result>
```

```xml Transaction already processed
<result>
  <status>TransactionAlreadyProcessed</status>
</result>
```

- **Validation error:** Show the message to customer
- **Incorrect operation invocation:** Check with the manager
- **Maestra unavailable:** Check with the manager and/or repeat request later.
- **Transaction already processed:** A transaction with this UUID was already processed. If you maintain a request queue and repeat a request until it's processed, it's a normal scenario. If you don't usually repeat requests, this response means that you've provided a non-unique transaction ID.
